# ADR 002 — Astronomer Cosmos over BashOperator for dbt Orchestration

**Date**: 2026-01-20
**Status**: Accepted

## Context

Airflow needs to orchestrate the dbt pipeline. Three common patterns exist:

| Pattern | Description |
|---|---|
| **BashOperator** | Single task runs `dbt run && dbt test`; entire pipeline is one Airflow node |
| **dbt Cloud + TriggerDagRunOperator** | Delegates execution to dbt Cloud; requires paid subscription |
| **Astronomer Cosmos** | Renders each dbt model as an individual Airflow task from the manifest |

## Decision

Use **Astronomer Cosmos** with `LoadMode.DBT_MANIFEST`.

The manifest is pre-generated by `dbt compile` before the DAG runs. Cosmos reads
it to build the task graph — one `.run` and one `.test` task per model — without
calling `dbt ls` on every scheduler heartbeat.

## Consequences

- **Positive**: Full task-level visibility in the Airflow UI. Failed models are
  immediately identifiable without parsing dbt logs.
- **Positive**: Airflow retries, SLAs, and callbacks apply at the model level,
  not the whole pipeline.
- **Positive**: Subset DAGs (`tickit_dbt_staging`, `tickit_dbt_marts`) use
  `RenderConfig(select=[...])` to run only the relevant part of the graph.
- **Negative**: `manifest.json` must be regenerated (`dbt compile`) after any
  model or schema change, or the task graph will be stale. This is a manual step
  before deployment.
- **Negative**: Cosmos adds ~0.5s per model at scheduler parse time. For very
  large projects (500+ models) this becomes noticeable. `LoadMode.DBT_MANIFEST`
  mitigates this compared to `LoadMode.DBT_LS`.
- **Trade-off vs BashOperator**: BashOperator is simpler and sufficient for small
  pipelines. Cosmos was chosen because individual task observability and retry
  granularity are worth the added complexity at this scale.
